SDWebImage底层实现原理

用请求图片的地址作为唯一标示符

缓存分为内存缓存和磁盘缓存 MemoryCache DiskCache
内存缓存默认空间大小512K，磁盘缓存默认空间大小10M

  1. 先用placeholderImage替代图片展示
  2. 根据imageUrl在本地查找图片，首先根据CacheKey查找是否在内存中。
  3. 如果内存中已经有缓存图片，将回调返回图片，展示图片。
  4. 如果内存中没有缓存图片，将生成NSInvocationOperation队列，会通过MD5(为了防止缓存的图片名称冲突)处理过的key从本地磁盘查找图片是否已经被下载缓存。
  5. 如果从本地磁盘获取到图片，将图片添加到内存缓存中，如果空闲内存过小，先清空内存缓存，然后展示图片。
  6. 如果没有从本地磁盘中获取到图片，说明所有缓存都不存在图片，需要下载图片。
  7. 图片下载由NSURLSession来进行，下载完成之后会保存到内存和硬盘。
  8. 进行界面展示。

优化TableView思路

tableview 流畅度可以用fps来测试，到60帧说明你优化tableView 已经很有经验了。

TableView的卡顿主要由下面几个方面造成：

  • 在下面方法中，频繁的创建cell 上的子控件并且添加到cell 上，这是一个要注意的地方，因为这样频繁的创建控件和添加会增加CPU的消耗，间接掉帧。

    解决方式：
    自定义Cell

  • tableView 高度问题。tableView 会频繁的调用如下方法，来先确定它的contentSize及每个Cell的位置，然后再调用cellForRowAtIndexPath 来显示。

    解决方式：
    在后台计算好高度以及布局，并缓存到内存重复使用。后台计算cell 的高度然后放到集合里面下次继续使用。

  • 图片圆角，阴影等操作，会引起离屏渲染。对CPU性能消耗。

    解决方式：
    处理图片圆角的时候将离屏渲染改成on-screen-rendering，节省消耗。给Image添加圆角

  • 有一些显示的内容有富文本，特别是从HTML 转化为属性字符串时候。

    解决方式：
    后台提前转化需要的属性字符串，然后缓存起来避免重复转化带来的CPU性能消耗。可以参考DTCoreText从HTML转化属性字符串的思路，他就是GCD后台转化的。

  • 一些显示很多图片的地方，服务器的图片很大，不是你控件的大小。

    解决方式：
    服务器返回控件宽高的图片，比如七牛可以在图片路径拼接参数来获取指定宽高。

  • 视图层次复杂情况下，CPU把它们混合会很消耗资源

    解决方式：
    如果不能避免，可以把你的视图绘制成一张图片来显示，当然渲染的过程在后台。可以参考VVeboTableViewDemo的思路。

  • 滑动了很多个Cell之后会加载很多没有显示的图片

    解决方式：TableView中实现平滑滚动延迟加载图片
    利用CFRunLoopMode的特性，可以将图片的加载放到NSDefaultRunLoopMode的mode里，这样在滚动UITrackingRunLoopMode这个mode时不会被加载而影响到。

IOS缓存机制详解

为什么要有缓存

应用需要离线工作的主要原因就是改善应用所表现出的性能。将应用内容缓存起来就可以支持离线。我们可以用两种不同的缓存来使应用离线工作。第一种是按需缓存，这种情况下应用缓存起请求应答，就和Web浏览器的工作原理一样；第二种是预缓存，这种情况是缓存全部内容（或者最近n条记录）以便离线访问。

像第14章中开发的Web服务应用利用按需缓存技术来改善可感知的性能而不是提供离线访问。离线访问只是无心插柳的结果。Twitter和Foursquare就是很好的例子。这类应用得到的数据通常很快就会过时。对于一条几天前的推文或者朋友上周在哪里你能有多大兴趣？一般来说，一条推文或者一条签到的信息只在几个小时内有意义，而24小时之后就变得无关紧要。不过大部分Twitter客户端还是会缓存推文，而Foursquare的官方客户端在无网络连接的情况下打开，会显示上次的状态。

大家可以用自己喜欢的Twitter客户端来试一下，Twitter for iPhone、Tweetbot或其他应用：打开某个朋友的个人资料并浏览他的时间线。应用会获取时间线并填充页面。加载时间线时会看到一个表示正在加载的圆圈在旋转。现在进入另一个页面，然后再回来打开时间线。你会发现这次是瞬间加载的。应用还是在后台刷新内容（在上次打开的基础上），但是它会显示上次缓存的内容而不是无趣地转圈，这样看起来就快多了。如果没有缓存，用户每次打开一个页面都会看到圆圈在旋转。无论网络连接快还是慢，减小网络加载慢的影响，让它看起来很快，是iOS开发者的责任。这就能大大改善用户满意度，从而提高了应用在App Store中的评分。

另一种缓存更加重视被缓存数据，并且能快速编辑被缓存的记录而无需连接到服务器。代表应用包括Google Reader客户端，稍后阅读类的应用Instapaper等。

NSOperation

不论是使用NSOperation还是使用GCD，都是为了程序能够并发的执行，提高运算效率。

首先我们要了解一个并发编程的几个概念：

  • 串行(Serial) && 并行(Concurrent)

串行和并行描述的是任务和任务之间的执行方式. 串行是任务A执行完了任务B才能执行, 它们俩只能顺序执行. 并行则是任务A和任务B可以同时执行.

  • 同步(Synchronous) && 异步(Asynchronous)

同步和异步描述的其实就是函数什么时候返回. 比如用来下载图片的函数A: {download image}, 同步函数只有在image下载结束之后才返回, 下载的这段时间函数A只能搬个小板凳在那儿坐等… 而异步函数, 立即返回. 图片会去下载, 但函数A不会去等它完成. So, 异步函数不会堵塞当前线程去执行下一个函数!

  • 并发(Concurrency) && 并行(Parallelism)

并发是程序的属性(property of the program), 而并行是计算机的属性(property of the machine).

解释一下, 并发和并行都是用来让不同的任务可以”同时执行”, 只是并发是伪同时, 而并行是真同时. 假设你有任务T1和任务T2(这里的任务可以是进程也可以是线程)。第一种情况：你的CPU是单核的，为了提高CPU的利用率，会将T1和T2“同时”执行，这里用的是分时执行，即CPU执行一会儿T1后马上再去执行T2，切换的速度非常快，（这里的切换也是需要消耗资源），造成的感觉是T1和T2同时执行（其实同一时刻只有一个任务占有着CPU）。第二种情况：你的CPU是多核的，这时就可以真正同时执行T1和T2了，在同一时刻CPU的核心core1执行T1，core2执行这T2。而且每个核心都还可以分时执行。

其实我们所编写的代码不能保证会被并行执行，但是我们可以并发的方式设计代码，系统会判断某一时刻是否有空闲的核心core，有的话就并行执行，没有空前的核心时就会在某个核心上分时并发执行。

RunLoops

作用：
  • 使程序一直运行并接受用户输入
  • 决定程序在何时应该处理那些Event
  • 调用解耦(Message Queue)
  • 节省CPU时间
Run Loops in Cocoa
  1. Foundation:NSRunLoop
  2. Core Foundation:CFRunLoop
  3. System:GCD、mach hernel、block、pthread…
    常用类：
    NSTimer、UIEvent、Autorelease、NSObject(NSDelayedPerforming、NSThreadPerformAddition)、CADisplayLink、CATransition、CAAnimation、dispatch_get_main_queue()、NSPort、NSURLConnection、AFNetworking

构成元素：

CFRunLoopTimer

CFRunLoopSource

  • Source是RunLoop的数据源抽象类(protocol)
  • RunLoop定义了两个Version的Source
    1. Source0：处理APP内部时间、APP自己负责管理（触发）。如UIEvent、CFSocket
    2. Source1：由RunLoop和内核管理、Mach port驱动。如CFMachPort、CFMessagePort
  • 如有需要，可从中选择一种来实现自己的Source
  • 上一条基本不会发生

Topic:RunLoopObserver 与 Autorelease Pool

UIKit通过RunLoopObserver在RunLoop两次Sleep间对AutoreleasePool进行Pop和Push，将这两次Loop中产生的Autorelease对象释放

CFRunLoopMode

  • RunLoop在同一时间只能且必须在一种特定Mode下Run
  • 更换Mode时，需要停止当前Loop，然后重启新Loop
  • Mode是iOS App滑动顺畅的关键

  • 可以定制自己的Mode

  • NSDefaultRunLoopMode 默认状态，空闲状态

  • UITrackingRunLoopMode 滑动ScrollView时，自动切换到这种
  • NSRunLoopCommonMode

RunLoop与dispatch_get_main_queue()

  • GCD中dispatch到main queue的block被分发到main RunLoop执行

RunLoop的挂起与唤醒

空闲时刻暂停程序
RunLoop会发送一个等待消息，挂起。CFRunLoopServiceMachPort

  • 指定用于唤醒的mach_port端口
  • 调用mach_msg监听唤醒端口，被唤醒前，系统内核将这个线程挂起，停留在mach_msg_trap状态
  • 由另一个线程（或另一个进程中的某个线程）向内核发送这个端口的msg后，trap状态被唤醒，RunLoop继续开始干活。

可变字典中setObject:forKey:和setValue:forKey:的区别

  1. setObject：forkey：中value是不能够为nil的，不然会报错。
    setValue：forKey：中value能够为nil，但是当value为nil的时候，会自动调用removeObject：forKey方法
  2. setValue：forKey：中key的参数只能够是NSString类型，而setObject：forKey：的可以是任何类型
注意：setObject：forKey：对象不能存放nil要与下面的这种情况区分：

  • [imageDictionarysetObject:[NSNullnull] forKey:indexNumber];
    [NSNull null]表示的是一个空对象，并不是nil，注意这点

  • setObject：forKey：中Key是NSNumber对象的时候，如下：
    [imageDictionarysetObject:obj forKey:[NSNumber numberWithInt：10]];

注意：

上面说的区别是针对调用者是dictionary而言的。
setObject:forKey:方法NSMutabledictionary特有的,而
setValue:forKey:方法是KVC（键-值编码）的主要方法。

深拷贝&&浅拷贝

  • 执行copy后，不管原始值是可变的还是不可变的， 生成的对象一定是不可变的。
  • 执行mutableCopy后，不管原始值是可变的还是不可变的， 生成的对象一定是不可变的。
  • 深拷贝&&浅拷贝与copy和mutableCopy无关，与copy和mutableCopy中的实现方法有关。
  1. 浅拷贝纸复制对象的本身，对象里的属性、包含的对象不做复制
  2. 深拷贝则既复制对象本身，对象的属性也会复制一份。
  3. Foundation 框架中支持复制的类，默认是浅拷贝。

Objective - C 对象的深浅拷贝的区别就在于你对copyWithZone的不同实现

  • 深拷贝、浅拷贝和retain之间的总结

copy、mutabelCopy 和 retain 之间的关系

  1. Foundation 可复制的对象，当我们copy的是一个不可变的对象时，他的作用相当于retain （cocoa做得内存优化）
  2. 当我们使用mutabelCopy 时 ，无论源对象师傅可变，副本是可变的，并且实现了真正意义上得拷贝
  3. 当我们copy 的是一个可变对象时，副本对象是不可变的，同样实现了真正意义上的拷贝
